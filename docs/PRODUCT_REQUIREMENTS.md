# Debate Studio 产品需求（第一阶段）

## 目标与边界

Debate Studio 是仅供个人在 macOS 本机运行的实验工具，用于让正方、反方、主持人（及可选裁判）模型进行可观察、可保存的正式辩论。优先顺序：稳定的流程、清楚的记录与故障恢复、易扩展的模型接入、舒适的个人使用体验。

第一阶段不提供账户、云端同步、多用户、后台服务、自动联网搜索、完整 PDF/音视频处理、无限辩论或原生 Gemini/DashScope/MiMo 适配。

## 第一阶段用户能力

- 管理多个平台连接、模型配置和本地别名；优先使用 macOS Keychain 保存密钥。
- 获取模型列表（可失败）；始终允许手填 Model ID、Base URL 和协议。
- 为正方、反方、主持人选择模型，创建辩题、背景、立场和范围。
- 上传图片或粘贴文本；图片仅在所选模型能力支持时原样提交，否则在开始前给出明确错误。
- 用 MockAdapter 或 OpenAI Chat Completions 兼容接口运行固定流程，并展示流式内容。
- 暂停、继续、停止、重试失败轮次、跳过当前阶段、强制总结；所有动作进入明确的命令记录。
- 查看调用耗时、Token、估算费用和中文错误说明；保存、查看并导出 JSON / Markdown（不含密钥）。

## 非功能需求

- Electron 采用 `contextIsolation: true`、`nodeIntegration: false`；preload 仅暴露白名单 IPC，所有 IPC 输入经 Zod 校验。
- 每次模型调用都是可中止、可持久化、可重试的独立轮次；进程重启后可读取历史状态。
- UI 使用简体中文；代码、接口、路径使用英文。
- 不永久硬编码模型名称；预设仅提供起始信息，用户配置为准。

## 风险、歧义与取舍

| 事项 | 风险或歧义 | 第一阶段决定 |
| --- | --- | --- |
| “暂停” | 供应商通常无法暂停一个正在生成的 HTTP 流 | 暂停表示终止当前请求并保留已收文本；继续创建新的、带上下文的轮次。UI 必须说明这一点。 |
| OpenAI 兼容性 | 各供应商的流式字段、工具、视觉能力并不一致 | 仅保证文本流与基础图片消息；能力不匹配时拒绝执行并建议相应协议。 |
| 私有研究 | 不主动索取或推断服务商未提供的隐藏推理链 | 保存模型显式产出的研究计划、来源、笔记和摘要。服务商主动返回的 `reasoning_content` 可在当前运行中瞬时展示，但不持久化、记录或导出。 |
| 费用 | 价格与 usage 返回不稳定 | 优先使用服务端 usage；缺失时标记为“估算/未知”，不维护实时价格库。 |
| Keychain | 原生 Keychain 集成增加桌面依赖和打包复杂度 | 抽象 CredentialStore；首版实现 Keychain 优先和应用数据目录回退，回退需用户显式确认。 |
| 图片 | 不同兼容端点对 URL/base64 支持差异大 | 将图片作为资产记录；仅当 ModelCapabilities 明示支持且适配器支持时发送。 |
| 状态恢复 | 流中断时不知模型实际已生成多少内容 | 增量持久化流文本；轮次标为 interrupted，可从该轮重试，不能假定服务端可续传。 |

## 待确认事项

1. 第一阶段 Keychain 的具体库选择：可接受小型原生依赖，还是先仅做应用数据目录存储？
2. 固定辩论流程中的“双方质询”是各一轮，还是每方可配置多轮？建议默认各一轮。
3. OpenAI Chat 的第一版是否允许用户输入自定义兼容端点？建议允许，但在 UI 中标记为“自行负责兼容性”。
4. 导出的图片应嵌入、复制到导出目录还是仅保留引用？建议 JSON 保留资产元数据和本地相对引用，Markdown 显示可访问路径。
